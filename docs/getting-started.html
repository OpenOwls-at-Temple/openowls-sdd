Getting Started with
OpenOwls SDD

A step-by-step guide to using AI coding assistants to build real projects — the OpenOwls way.

Section 01

The OpenOwls SDD File Structure

Every OpenOwls project follows this layout. The two files at the root level are read first by Claude Code. Everything inside ai_specs/ provides the full project context — what to build, how to build it, and how to ship it.

project-root/
├── CLAUDE.md ← tells Claude Code to read everything below, in order
├── progress.md ← living session status, updated after every work session
└── ai_specs/
    ├── overview.md ← project goals, users, tech stack
    ├── features.md ← Phase 1 / 2 / 3 feature definitions
    ├── architecture-planning.md ← folder structure, design decisions, API
    ├── domain-knowledge.md ← terminology, business rules, constraints
    ├── conventions.md ← coding style, git rules, testing requirements
    └── deployment.md ← hosting, environment variables, deploy steps
CLAUDE.md
Root · Claude Code Entry Point

Tells Claude Code to read all ai_specs/ files in the correct order before doing any work. Also defines global behavior rules.

progress.md
Root · Session Memory

Tracks what is done, in progress, and blocked. Read first every session since Claude Code has no memory between sessions.

ai_specs/overview.md
Audience: Business Sponsor

Project name, problem statement, goals, non-goals, target users, tech stack, and key constraints.

ai_specs/features.md
Audience: End User / Product Owner

All features organized into Phase 1 (MVP), Phase 2 (enhanced), and Phase 3 (AI/advanced), with acceptance criteria.

ai_specs/architecture-planning.md
Audience: Architect / Engineer

Folder structure, design decisions, data models, API endpoints, LLM integration plan, and environment variables.

ai_specs/domain-knowledge.md
Audience: AI Assistant + Team

Domain terminology, business rules, common pitfalls, and external dependencies the AI needs to know upfront.

ai_specs/conventions.md
Audience: Engineer / AI Assistant

Naming conventions, code style, git branch rules, testing requirements, and rules for Claude Code behavior.

ai_specs/deployment.md
Audience: DevOps / Engineer

Hosting platforms, local setup steps, environment variables, CI/CD pipeline, and common deployment issues.

Section 02

Who Creates Each File

OpenOwls SDD mirrors real professional software development — different stakeholders own different documents. Faculty seed the high-level vision; students own the technical execution.

File Created By Notes
overview.md Faculty Sets project scope and ensures feasibility before students begin
features.md Faculty + Students Faculty defines Phase 1 (MVP). Students define Phase 2 and Phase 3 after Phase 1 ships.
architecture-planning.md Students Core technical learning exercise — students make and justify design decisions
domain-knowledge.md Faculty + Students Faculty seeds domain concepts. Students research and expand throughout the project.
conventions.md Faculty Provided as a template to ensure consistency across all student projects
deployment.md Students Students choose hosting platforms and document the full deployment process
progress.md Students Updated at the end of every work session throughout the project lifecycle
Section 03

The Four-Step Workflow

Every project follows the same ritual. Claude.ai handles planning. Claude Code handles building. The ai_specs/ files are the handoff contract between the two.

  1. Project Initialization — Claude.ai

    Upload the OpenOwls SDD template files to Claude.ai and describe your project idea. Claude.ai generates a complete, filled-in set of ai_specs/ files tailored to your project. Faculty reviews overview.md and features.md before handing off to students.

    Example prompt to Claude.ai

    "These are my OpenOwls SDD template files. I want to build [describe your project]. The target users are [describe users]. Please generate a complete filled-in set of ai_specs/ files following the same structure as the templates."

    Tip: Generate one file at a time. Ask Claude to produce overview.md first, review it, then move to features.md. This gives you more control over the output.

  2. Faculty Review

    Before students touch any code, faculty reviews overview.md and features.md to confirm the project is well-scoped, Phase 1 is a realistic MVP, and the tech stack is appropriate.

    Key question: Can a team of students realistically complete Phase 1 in two weeks? If not, move features to Phase 2.

  3. Students Complete the Specs

    Students take ownership of architecture-planning.md and deployment.md, making real technical decisions about how to build and ship the project. They expand domain-knowledge.md as they learn the problem space. All files are checked into the GitHub repository.

    This step is the learning. Students who skip or rush the spec files will struggle during implementation. The planning process is not overhead — it is the work.

  4. Implementation — Claude Code

    With the ai_specs/ files complete, students open Claude Code in the project repository. Claude Code reads CLAUDE.md automatically and loads all spec files. Students work phase by phase, updating progress.md at the end of every session.

    Example Claude Code session opener

    "Read all files in ai_specs/ and progress.md. We are currently in Phase 1. The next task is [task from progress.md]. Let's begin."

Section 04

Golden Rules for Students

Always update progress.md after every session

Claude Code has no memory between sessions. If you do not update progress.md, the next session starts blind. Move completed items to done, note what is in progress, and log any blockers.

Stay inside the current phase

Do not ask Claude Code to implement Phase 2 features while Phase 1 is incomplete. Phasing exists to keep scope under control and ensure you have a working product at every milestone.

Never commit secrets to the repository

API keys and database passwords must never appear in any committed file. Use a .env file locally and add it to .gitignore. The deployment.md file documents which variables are needed — not their values.

The spec files are the source of truth

If Claude Code does something unexpected, check whether your spec files are clear enough. Ambiguity in docs leads to wrong assumptions in code. Update the spec file first, then re-run the task.

Review before you accept

Claude Code is a collaborator, not an authority. Read every diff before accepting it. If generated code contradicts your conventions.md or architecture-planning.md, say so and ask Claude to correct it.

Section 05

Understanding the Three Phases

Every OpenOwls project is scoped into three phases. This forces students to define a real MVP before adding complexity — one of the most important skills in professional software development.

PhaseGoalDefinition of Done
Phase 1 Core MVP — the smallest useful version of the app A real user can complete the primary workflow end to end
Phase 2 Enhanced features — meaningful value on top of a stable Phase 1 Phase 1 is fully tested and deployed before Phase 2 begins
Phase 3 Advanced / AI features — LLM-powered capabilities or stretch goals Phase 2 is stable; AI integration is thoughtful and well-prompted
Faculty note: The most common student mistake is a Phase 1 that is too large. A good Phase 1 is one feature, working end-to-end, deployed and usable by a real person. Everything else is Phase 2.
Section 06

Motivation and Rationale

OpenOwls SDD is a deliberate simplification of industry Spec-Driven Development practices — adapted for a pedagogical context. This section explains the thinking behind the framework and why it is structured the way it is.

Each file mirrors a real professional role

The most important design principle behind OpenOwls SDD is that every specification file maps to a real stakeholder in a professional software project. Students are not just filling in templates — they are inhabiting real roles that exist in the industry.

File Professional Equivalent What It Teaches
overview.md Business sponsor defining the product vision How to articulate a problem worth solving and scope a project realistically
features.md Product owner translating user conversations into requirements How to write user stories, define acceptance criteria, and control scope with phasing
architecture-planning.md System architect designing the technical approach How to make and justify design decisions before writing a single line of code
domain-knowledge.md Domain expert briefing the engineering team How to research a problem space and surface the constraints that shape implementation
conventions.md Engineering lead setting team standards Why consistency matters across a codebase and how to enforce it
llm-integration.md AI/ML engineer designing the intelligence layer How to think rigorously about prompts, context, evaluation, and failure modes
deployment.md DevOps engineer owning the release pipeline That shipping is part of building — deployment is not an afterthought
progress.md Engineering team's daily standup and sprint tracker How to maintain continuity across sessions and communicate project state clearly

Why this is a simplification — and why that is intentional

Industry Spec-Driven Development frameworks like GitHub Spec Kit are more granular — they produce per-feature spec files, detailed task breakdowns, and tight CI/CD integration. OpenOwls SDD deliberately consolidates these into fewer, broader files for two reasons.

First, cognitive load. A student encountering SDD for the first time should not be overwhelmed by a dozen file types before they have written a line of code. OpenOwls SDD is designed to be learnable in a single onboarding session.

Second, audience clarity. Each OpenOwls SDD file has a named human audience — a business sponsor, an end user, a system architect. This stakeholder framing is absent from most industry frameworks, which assume the reader is already an engineer. For students, naming the audience is a teaching device that builds empathy and communication skills alongside technical ones.

Why specs before code

The single most common failure mode in student projects is starting to code before the problem is well understood. Features grow unchecked, the architecture becomes inconsistent, and the project collapses under its own complexity before Phase 1 is complete.

Writing the ai_specs/ files first forces a team to confront hard questions early — what exactly are we building, for whom, and what does done look like? — when the cost of changing course is low. By the time Claude Code opens, the team has already made the decisions that matter most.

Why the LLM is a first-class concern

Every OpenOwls project has an LLM layer. This is not incidental — it is the mission. OpenOwls exists to teach students how to build AI-powered tools, not just web applications. The llm-integration.md file exists because prompt design, context management, error handling, and evaluation are engineering disciplines that deserve the same rigor as any other part of the system. Treating the LLM as an afterthought produces brittle, unpredictable applications. Treating it as a first-class component produces ones that work.

Relationship to industry SDD

OpenOwls SDD is inspired by, but not identical to, the industry Spec-Driven Development movement. The ai_specs/ folder name signals this distinction deliberately — it is the OpenOwls convention, not a direct adoption of GitHub Spec Kit or any other framework. Students who go on to work in industry will encounter those frameworks and find them familiar. The core habits — write specs first, keep them in version control, use them to brief your AI tools — are the same. The structure is simplified for the classroom.

Relationship to Computer Science Software Engineering Curriculum

OpenOwls SDD may also serve as a starting point for a broader conversation about how Software Engineering is taught. Traditional CS programs rightly emphasize programming, data structures, and algorithms — the technical foundation every engineer needs. OpenOwls SDD complements that foundation by giving students early exposure to the design phase of software development and to the perspectives of the different stakeholders involved in building a real product. Writing specs before code, thinking like a product owner, an architect, and a domain expert — these are skills that make a stronger, more well-rounded engineer. The framework is modest in scope, but the habits it builds carry well beyond the classroom.
