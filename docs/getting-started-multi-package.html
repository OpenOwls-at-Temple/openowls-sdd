OpenOwls · SDD Suite Edition

Spec-Driven Development for a
Multi-Package Monorepo

The advanced, two-tier version of OpenOwls SDD — for a suite of stacked libraries that live and build together in one repository.

Section 01

What You're Building

The original OpenOwls SDD guide assumes one stand-alone application. This Suite Edition is for the next step up: a suite of libraries, not a single app.

Several Python packages are stacked on top of one another — each depends on the one below. Every package keeps its own .venv and its own requirements.txt, yet they all work together and live in one monorepo. Dependencies point downward only. Each package is a top-level folder at the suite root — there is no wrapping packages/ directory.

package3

Top layer — user- and model-facing. Talks to people and the LLM. Assembled on top of everything below.

package2

Middle layer — orchestration. Built on package1; consumes it only through its public API.

package1

Section 02

The Two-Tier Spec Structure

The key idea: specs live at two levels. A shared layer at the root that every package inherits, and a private layer inside each package. Operational files — CLAUDE.md and the progress files — sit loose at each directory root; everything that is a spec goes inside an ai_specs/ folder.

Tier 1 — Shared, system-wide (the root)
your-suite/
├── CLAUDE.md ← system entry point — read first (loose)
├── project_progress.md ← stack-wide rollup + index to each package's progress (loose)
└── ai_specs/ ← SHARED, system-wide specs
    ├── overview.md ← the whole stack's purpose
    ├── domain-knowledge.md ← shared terminology & business rules
    ├── conventions.md ← one shared style / git / testing standard
    ├── llm-integration.md ← shared model choice & guardrails
    ├── auth-security.md ← shared identity model & security posture
    └── architecture-planning.md ← THE layering: package order & dependency direction
Tier 2 — Private, per package (same shape for every package)
package1/ ← a package — a top-level folder, NOT nested under packages/
├── CLAUDE.md ← package entry: "read ./ai_specs/, inherit root" (loose)
├── progress.md ← this library's session memory (loose)
├── ai_specs/
│   ├── overview.md ← what THIS library does, its API, who consumes it
│   ├── features.md ← this library's capabilities, phased
│   ├── architecture.md ← this library's internal design
│   └── deployment.md ← how this library is packaged & versioned
├── .venv/ ← package-owned environment (git-ignored)
└── requirements.txt
package2/ ← another package — same shape, also a top-level folder
└── … (add one folder per package: package3, …)
Structural invariant: CLAUDE.md and the progress files are operational and must sit at the folder root — never inside ai_specs/. Claude Code discovers CLAUDE.md by walking up the directory tree, so it has to live where you work and point into ai_specs/.
Section 03

What Goes Where — and Why

Each spec is placed at the level where it is naturally owned. Shared policy lives once at the root; package-specific detail lives with the package. This is the decision log.

Spec Level Decision & Rationale
conventions.md Root One standard for all packages; copying it per-package guarantees drift the moment a rule changes.
domain-knowledge.md Root Shared vocabulary and business rules for the whole team.
architecture-planning.md Root The layering — which package depends on which, in which direction — is inherently between packages and can't live in any one of them.
llm-integration.md Root Shared policy (model choice + guardrails). Caveat: if only the top package calls the model, move prompt specifics into that package.
auth-security.md Root Shared policy (identity model + posture). Caveat: implementation specifics can move to the package that owns the boundary.
overview.md Both Root = the whole stack's purpose. Package = what this library does, its API, its consumers. Same name, different scope — complementary, not duplicated.
features.md Package Each library has its own phased capabilities.
architecture.md Package The within-package design. Distinct from the root's between-package layering.
deployment.md Package How that specific library is packaged and versioned.
progress.md Package Per-library session memory.
project_progress.md Root Stack-wide rollup; an index pointing to each package's progress.md.
Section 04

How Claude Code Reads the Monorepo

The two-tier structure isn't arbitrary — it mirrors exactly how Claude Code discovers and loads context. Understanding these mechanics is what makes the design click.

Root specs load by walking up

From wherever Claude Code launches, it walks up to the repo root and reads every CLAUDE.md it finds. So the root CLAUDE.md and the shared specs it points at are always in context, in every package. This is precisely why shared standards belong at the root — they're inherited for free.

Package specs load on demand

A nested CLAUDE.md is not loaded at launch. It's pulled in the first time Claude Code reads a file in that subtree. This is the focus mechanism — a package's detail arrives only when you're actually working in it.

Launch inside a package for focus

Running cd package1 && claude means Claude Code still inherits the root rules (walk-up), but its attention and on-demand loading center on that one library.

Example focused session opener

"Read ./ai_specs/ and ./progress.md for this package, and follow the root CLAUDE.md conventions. We are in Phase 1 of package1. The next task is [task from progress.md]. Let's begin."

Keep instruction files concise

Instruction files over ~200 lines consume more context and can reduce how reliably they're followed. Keep each CLAUDE.md short and let it reference the ai_specs/ files rather than restate them.

Section 05

Per-Package Environment Rules

Each package owns its own environment, and each package's CLAUDE.md is the home for its environment instructions. Because that file loads on demand, the right rules arrive exactly when Claude Code starts working in that library.

Example rule in a package's CLAUDE.md

"Before running, testing, or installing anything in this package: activate ./.venv. Install dependencies only from ./requirements.txt into that venv. Never activate or modify a sibling package's environment."

One venv per package, never cross-activated. A leaked or shared environment is one of the easiest ways to make a "works on my machine" bug that only appears once another layer is built on top.
Section 06

Build Order — Bottom-Up

In a stacked suite, dependency order is a phasing discipline. You build from the foundation up, and you don't climb the stack until the layer below is solid.

  1. package1 (base) first

    Build and stabilize the base. Its own tests pass and its public API is settled before anything depends on it. A churning foundation forces rework in every layer above.

  2. package2 (middle) next

    Only once package1 is stable. Consume package1 through its public API — never reach around it into internals.

  3. package3 (top) last

    The user- and model-facing layer, assembled on a foundation that already works end-to-end.

Within each library, a real Phase 1 is the smallest version that works end-to-end; everything else is Phase 2. The most common mistake is starting the top package because it's exciting while the base (package1) is still churning.
Section 07

Golden Rules for the Suite

Shared specs live once, at the root

Never copy a root spec into a package. A single source of truth is the whole point of the two-tier structure.

One venv per package, never cross-activated

Activate only the current package's ./.venv, and install only from its ./requirements.txt. Leave sibling environments alone.

Log work in two places

Update the package's progress.md and keep the root project_progress.md current. The rollup is how anyone sees the whole stack's state at a glance.

Stay in your layer

Respect the downward-only dependency direction. A higher layer may use a lower one through its public API; a lower layer must never depend on a higher one.

Specs are the source of truth

If Claude Code does something unexpected, fix the relevant spec first — root or package — then re-run the task. Ambiguity in docs becomes wrong assumptions in code.

Never commit secrets

Use .env + .gitignore. For a stack that shares one LLM provider key, a single root .env is usually simpler than one per package.
